## Optimization Strategy

### 1. Self-Hosted PostgreSQL on EC2 (Save $49.50/month)

**Current:** RDS db.t3.medium ($79.50/month) + storage/backups ($19.50/month) = $99/month

**Optimized:** Self-hosted on existing EC2 instance

#### 1.1 Use Existing Processing Instance

**Option A: Run PostgreSQL on Processing Instance (RECOMMENDED)**
- Already running EC2 instance for Flask API
- Add PostgreSQL to same instance (minimal CPU overhead)
- Current instance: Likely t3.medium or larger
- PostgreSQL + Flask can coexist efficiently

**Instance Costs:**
- t3.medium: $0.0416/hour = $30.24/month
- EBS gp3 100GB: $0.08/GB/month = $8/month
- **Total: $38/month** (vs $99/month RDS)
- **Savings: $61/month**

**But wait:** Processing instance already exists for Flask API, so PostgreSQL is effectively FREE (only storage cost = $8/month)

**Actual Savings Calculation:**
- RDS total: $99/month
- Self-hosted incremental cost: $8/month (EBS only, instance already paid for)
- **NET SAVINGS: $91/month**

**Why This Works:**
- Flask API + PostgreSQL both use minimal CPU
- PostgreSQL RAM: ~1-2GB (fits in t3.medium 4GB RAM)
- Flask RAM: ~500MB
- Total RAM usage: ~2.5GB (safe margin on 4GB instance)

#### 1.2 Installation Steps (Not Executed - Planning Only)

**Step 1: Install PostgreSQL on Processing Instance**
```bash
# SSH to Processing Instance
ssh -i /home/ploxyz.pem ubuntu@PROCESSING_IP

# Update system
sudo apt-get update

# Install PostgreSQL 14
sudo apt-get install -y postgresql-14 postgresql-contrib-14

# Start and enable
sudo systemctl start postgresql
sudo systemctl enable postgresql

# Verify
sudo systemctl status postgresql
```

**Step 2: Configure PostgreSQL**
```bash
# Edit PostgreSQL config for performance
sudo nano /etc/postgresql/14/main/postgresql.conf

# Recommended settings for t3.medium (4GB RAM):
shared_buffers = 1GB                # 25% of RAM
effective_cache_size = 3GB          # 75% of RAM
maintenance_work_mem = 256MB
checkpoint_completion_target = 0.9
wal_buffers = 16MB
default_statistics_target = 100
random_page_cost = 1.1              # For SSD
effective_io_concurrency = 200
work_mem = 10MB
min_wal_size = 1GB
max_wal_size = 4GB

# Restart PostgreSQL
sudo systemctl restart postgresql
```

**Step 3: Create Database and User**
```bash
sudo -u postgres psql

CREATE DATABASE poker_data;
CREATE USER poker_user WITH PASSWORD 'SECURE_PASSWORD_HERE';
GRANT ALL PRIVILEGES ON DATABASE poker_data TO poker_user;

# Enable remote connections (if needed)
\q

# Edit pg_hba.conf
sudo nano /etc/postgresql/14/main/pg_hba.conf

# Add line:
host    poker_data    poker_user    127.0.0.1/32    md5

# Edit postgresql.conf for local connections
sudo nano /etc/postgresql/14/main/postgresql.conf

# Change:
listen_addresses = 'localhost'

# Restart
sudo systemctl restart postgresql
```

**Step 4: Run Schema DDL**
```bash
# Copy schema.sql to instance
scp -i /home/ploxyz.pem ~/POKERBET_SQL_SCHEMA.sql ubuntu@PROCESSING_IP:~/

# Run schema
psql -U poker_user -d poker_data -f ~/POKERBET_SQL_SCHEMA.sql

# Verify tables
psql -U poker_user -d poker_data
\dt
\di  # List indexes
```

**Step 5: Update Flask API Connection**
```python
# Edit app.py
DB_CONFIG = {
    'host': 'localhost',  # Changed from RDS endpoint
    'database': 'poker_data',
    'user': 'poker_user',
    'password': os.environ.get('DB_PASSWORD')  # Use env var for security
}
```

#### 1.3 Manual Backup Strategy (Replace RDS Automated Backups)

**Daily Backup Script:**
```bash
#!/bin/bash
# /opt/scripts/backup_postgres.sh

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/opt/backups/postgres"
S3_BUCKET="poker-backups"

mkdir -p $BACKUP_DIR

# Full database dump
pg_dump -U poker_user -d poker_data | gzip > $BACKUP_DIR/poker_data_$DATE.sql.gz

# Upload to S3
aws s3 cp $BACKUP_DIR/poker_data_$DATE.sql.gz s3://$S3_BUCKET/postgres/$DATE.sql.gz

# Keep local backups for 7 days
find $BACKUP_DIR -name "*.sql.gz" -mtime +7 -delete

echo "Backup complete: $DATE"
```

**Cron Schedule:**
```bash
# Daily at 2 AM
0 2 * * * /opt/scripts/backup_postgres.sh >> /var/log/postgres_backup.log 2>&1
```

**Backup Costs:**
- S3 storage for backups: ~100GB/month compressed = $2.30/month
- Already included in S3 cost estimate

**Restore Procedure:**
```bash
# Download backup from S3
aws s3 cp s3://poker-backups/postgres/20260406_020000.sql.gz .

# Restore
gunzip poker_data_20260406_020000.sql.gz
psql -U poker_user -d poker_data < poker_data_20260406_020000.sql
```

### 2. Aggressive S3 Glacier Transition (Save $50/month)

**Current:** 90-day Standard → Glacier
- 2.3TB/month new data
- Average age: 45 days in Standard
- Cost: ~$150/month

**Optimized:** 30-day Standard → Glacier
- Same 2.3TB/month new data
- Average age: 15 days in Standard
- Cost: ~$100/month
- **Savings: $50/month**

#### 2.1 S3 Lifecycle Policy

**Update Lifecycle Rule:**
```json
{
  "Rules": [
    {
      "Id": "ArchiveRawPokerData",
      "Status": "Enabled",
      "Filter": {
        "Prefix": ""
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "GLACIER_INSTANT_RETRIEVAL"
        }
      ],
      "NoncurrentVersionTransitions": [
        {
          "NoncurrentDays": 30,
          "StorageClass": "GLACIER_INSTANT_RETRIEVAL"
        }
      ]
    }
  ]
}
```

**Apply via AWS CLI:**
```bash
# Save policy to file
cat > lifecycle-policy.json << 'EOF'
{
  "Rules": [
    {
      "Id": "ArchiveRawPokerData",
      "Status": "Enabled",
      "Filter": {"Prefix": ""},
      "Transitions": [
        {"Days": 30, "StorageClass": "GLACIER_INSTANT_RETRIEVAL"}
      ]
    }
  ]
}
EOF

# Apply to bucket
aws s3api put-bucket-lifecycle-configuration \
  --bucket poker-raw-data \
  --lifecycle-configuration file://lifecycle-policy.json

# Verify
aws s3api get-bucket-lifecycle-configuration --bucket poker-raw-data
```

#### 2.3 Glacier Instant Retrieval vs Standard Glacier

**Why Glacier Instant Retrieval?**
- Retrieval time: Milliseconds (vs 3-5 hours for Standard Glacier)
- Cost: $0.004/GB/month (vs $0.0036/GB for Standard Glacier)
- Use case: May need to reprocess old data quickly

**Trade-off:**
- Slightly more expensive than Standard Glacier ($0.0004/GB difference)
- For 100TB: Extra cost = $40/month
- Worth it for instant access to audit trail

**Alternative (More Aggressive):**
```json
{
  "Transitions": [
    {"Days": 30, "StorageClass": "GLACIER_DEEP_ARCHIVE"}
  ]
}
```
- Cost: $0.00099/GB/month (75% cheaper than Glacier Instant)
- Retrieval: 12-48 hours
- **Additional savings: ~$300/month at 100TB scale**
- Only use if you NEVER need to reprocess old data quickly

#### 2.4 Retrieval Cost Considerations

**Glacier Instant Retrieval Pricing:**
- Storage: $0.004/GB/month
- Retrieval: $0.03/GB (first 10TB/month free tier?)
- GET requests: $0.01 per 1,000 requests

**Example Reprocessing Scenario:**
- Reprocess 1 day of data: ~77GB
- Retrieval cost: 77GB × $0.03 = $2.31 (one-time)
- Acceptable for debugging/auditing

**Budget for Retrievals:**
- Assume 1 reprocessing event per month
- Cost: ~$3/month

## Risk Mitigation

### Risk 1: Data Loss (Self-Hosted PostgreSQL)

**Risk:** No automated RDS backups, rely on manual scripts

**Mitigation:**
- Daily automated backups to S3
- Retain 30 days of backups
- Test restore procedure monthly
- Monitor backup script execution
- Alert on backup failures

**Backup Redundancy:**
- S3 Standard-IA: Primary backup storage
- Glacier Deep Archive: Long-term retention (after 90 days)
- Multiple AZs in S3 (automatic)

### Risk 2: Performance Degradation

**Risk:** Self-hosted may be slower than managed RDS

**Mitigation:**
- Benchmark before/after migration
- Monitor query performance with pg_stat_statements
- Tune PostgreSQL config for workload
- Scale instance if needed (t3.medium → t3.large)

**Performance Targets:**
- Simple queries: <10ms (same as RDS)
- Join queries: <50ms
- Analytics: <500ms
- Dashboard: <100ms

**Monitoring:**
```sql
-- Weekly performance check
SELECT 
    query,
    mean_exec_time,
    calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

### Risk 3: Glacier Retrieval Delays

**Risk:** 30-day transition means more data in Glacier, slower access

**Mitigation:**
- Use Glacier Instant Retrieval (millisecond access)
- Not Glacier Deep Archive (12-48 hour delay)
- Budget for retrieval costs (~$3/month)
- Document retrieval procedures

**When You Need Old Data:**
```bash
# Glacier Instant Retrieval - immediate access
aws s3 cp s3://poker-raw-data/2026/03/01/snapshot.json . 
# Works immediately, no restore needed
```

### Risk 4: Instance Failure

**Risk:** Self-hosted on single EC2 instance (no HA)

**Mitigation:**
- Daily backups to S3 (restore in <30 minutes)
- EBS snapshots (automated weekly)
- Documented restore procedure
- Monitoring with CloudWatch alarms

**Recovery Time Objective (RTO):**
- Target: 30 minutes to restore from backup
- Maximum: 2 hours (including instance launch)

**Restore Procedure:**
```bash
# 1. Launch new instance (if needed)
aws ec2 run-instances --instance-type t3.medium ...

# 2. Download latest backup
aws s3 cp s3://poker-backups/postgres/latest.sql.gz .

# 3. Restore database
gunzip latest.sql.gz
psql -U poker_user -d poker_data < latest.sql

# 4. Restart Flask API
sudo systemctl restart flask-api

# 5. Verify
curl http://localhost:5000/api/health
```
